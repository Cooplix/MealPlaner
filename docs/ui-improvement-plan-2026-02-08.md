# План покращення UI/UX для MealPlaner

Дата: 2026-02-08

## 1. Мета
Покращити інтерфейс без радикальної зміни стилю:
- зберегти поточну візуальну мову (світла тема, акуратні картки, Tailwind),
- зробити інтерфейс більш зручним,
- зробити інтерфейс більш інформативним через кольори, стани та пріоритети.

## 2. Поточний стан
- UI функціональний, але місцями примітивний.
- Багато сторінок мають схожі блоки, але без єдиного стандарту для:
  - status badges,
  - повідомлень,
  - порожніх станів,
  - пріоритетності дій.
- Важливі дані часто мають однакову візуальну вагу, через що складніше швидко читати екран.

## 3. Принципи покращення
1. Зберегти знайому структуру, змінити якість подачі.
2. Кожен екран має відповідати на питання:
- Що важливо зараз?
- Що потрібно зробити наступним?
- Що вже під контролем?
3. Колір використовувати як функціональний сигнал, а не декор.
4. Будь-яка таблиця має мати:
- чітку ієрархію колонок,
- зрозумілі статуси,
- швидкі дії,
- читабельний empty/loading/error.

## 4. Колірна семантика (єдина для всього додатку)
- `green`: норма / completed / healthy.
- `blue`: info / neutral actionable.
- `amber`: attention / скоро потрібно діяти.
- `red`: critical / overdue / blocked.
- `gray`: secondary / metadata.

Правила:
- Статуси показувати не лише текстом, а й кольоровим бейджем або тоном блоку.
- Для accessibility завжди поєднувати колір + текст + іконку.

## 5. Стандарти компонентів

## 5.1 Нова базова бібліотека UI-елементів (в межах проєкту)
- `StatusBadge` (success/info/warn/error).
- `InfoCard` (title, value, delta, hint).
- `EmptyState` (іконка, причина, CTA).
- `InlineAlert` (success/warn/error).
- `SectionHeader` (title, subtitle, primary action, secondary action).
- `DataTableToolbar` (search, filters, count, export).

## 5.2 Єдині стани завантаження
- Skeleton для таблиць і карток.
- Loading indicator не повинен зміщувати layout.

## 5.3 Єдині стани помилки
- Короткий діагноз.
- Кнопка `Повторити`.
- Якщо можливо, порада для виправлення.

## 6. Інформативність по модулях

## 6.1 Календар
- Додати daily summary:
  - загальна калорійність дня,
  - покриття слотів (наприклад, 3/4),
  - warning якщо день неповний.
- Рекомендації:
  - бейджі причин рекомендації (`економія`, `запаси`, `різноманіття`).
  - візуальний score-рівень (high/medium/low fit).

## 6.2 Запаси
- Зробити індикатори пріоритету:
  - скоро закінчиться термін,
  - нижче min,
  - достатній запас.
- Додати компактні summary cards зверху:
  - прострочені,
  - закінчуються за 7/14 днів,
  - потребують поповнення.

## 6.3 Покупки
- Виділити ключові поля:
  - сума,
  - магазин,
  - дата,
  - товар.
- Додати quick-filters:
  - `сьогодні`, `7 днів`, `30 днів`, `цей магазин`.

## 6.4 Список покупок
- Показати прогрес виконання списку.
- Групування за категоріями.
- Маркери пріоритету (що терміново купити).

## 6.5 Витрати і собівартість
- Виділити top insights:
  - найбільші витрати,
  - найбільш подорожчалі позиції,
  - найдорожчі страви.
- Додати кольорові індикатори тренду (up/down/stable).

## 7. Юзабіліті покращення
- Уніфікувати primary/secondary/destructive кнопки.
- Зменшити когнітивне навантаження:
  - важливі дії зверху праворуч,
  - другорядні сховати в компактні меню.
- Клавіатурна доступність:
  - таб-навігація,
  - фокус-стили,
  - Enter/Escape в модалках.
- Мобільний режим:
  - sticky панель дій,
  - адаптовані таблиці (card-list для малих екранів).

## 8. Етапи реалізації

### Крок 1. UI audit і дизайн-специфікація
- Інвентаризація всіх поточних екранів.
- Фіксація pain points.
- Визначення token-стилів (кольори, відступи, статуси).

DoD:
- Є погоджений UI spec документ.
- Актуальний аудит зафіксовано в `docs/ui-audit-2026-02-09.md`.

### Крок 2. Design tokens і базові компоненти
- Додати спільні компоненти (`StatusBadge`, `InlineAlert`, `SectionHeader`, `EmptyState`).
- Замінити дубльовану розмітку на стандартизовану.

DoD:
- Всі ключові сторінки використовують однакові базові компоненти.

### Крок 3. Таблиці та toolbar стандарти
- Уніфікувати таблиці на `inventory`, `purchases`, `shopping`.
- Додати єдину панель фільтрів і лічильник записів.

DoD:
- Табличні екрани мають однакову UX-логіку і вигляд.

### Крок 4. Статуси і колірна інформативність
- Упровадити кольорові статуси у всіх модулях.
- Підтвердити контрастність і доступність.

DoD:
- Критичні/важливі стани читаються за 1-2 секунди без додаткових кліків.

### Крок 5. Календар і рекомендації
- Підсилити блок рекомендацій (бейджі причин, score).
- Додати day completeness summary.

DoD:
- Рекомендації не тільки показані, а й пояснені.

### Крок 6. Dashboard-патерни на сторінках аналітики
- Додати summary cards на витрати/собівартість/запаси.
- Додати тренди і виділення аномалій.

DoD:
- На кожній аналітичній сторінці є зрозумілий top-level summary.

### Крок 7. Мобільне і accessibility hardening
- Перевірка брейкпоінтів.
- Перевірка клавіатурної навігації і фокусу.

DoD:
- UX стабільний на desktop і mobile.

### Крок 8. UX smoke tests
- Написати smoke-тести на критичні UI flow.

DoD:
- Основні сценарії проходять без регресій.

## 9. Метрики успіху
- Скорочення часу на виконання базових дій:
  - додавання покупки,
  - планування тижня,
  - пошук критичних запасів.
- Збільшення використання рекомендацій.
- Зменшення помилок користувача в формах.
- Покращення суб'єктивної оцінки зручності (коротке опитування).
